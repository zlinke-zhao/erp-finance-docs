一、业务背景

1.1 问题描述

在与越南、泰国等海外供应商对账时，经常会出现品质扣款、迟延罚款等情况。传统做法是从应付账款中直接扣减，但这些国家的税法有特殊规定： > 核心约束：发票已开具、双方已对账确认的金额，不能再从应付账款中扣减。 因此，必须通过现金收/付款项的方式单独处理扣款项。

1.2 业务场景

| 场景 | 说明 | 扣款方向 | |------|------|----------| | 品质扣款 | 货物质量问题，供应商赔款 | 供应商→我方 | | 迟延罚款 | 供应商延期交付，被罚款 | 供应商→我方 | | 我方补偿 | 我方责任，主动赔偿供应商 | 我方→供应商 | | 其他补偿 | 价格调整、业务补偿等 | 双方均可 | ---

2.2 功能边界

本模块负责：
  • 现金扣款的申请、审核、确认
  • 与应付管理的联动
  • 凭证模板的生成
  • 往来核销处理
本模块不负责：
  • 采购对账逻辑（由采购模块处理）
  • 银行转账处理（由现金管理模块处理）
  • 发票开具/作废（由发票模块处理）

三、业务流程

3.1 完整流程

`` ┌─────────────────────────────────────────────────────────────────┐ │ 现金扣款处理完整流程 │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ ① 采购对账发现扣款项 │ │ ↓ │ │ ② 发起现金扣款申请 │ │ ↓ │ │ ③ 财务审核确认 │ │ ↓ │ │ ④ 生成现金扣款单 │ │ ↓ │ │ ⑤ 现金收/付款确认 │ │ ↓ │ │ ⑥ 往来核销处理 │ │ ↓ │ │ ⑦ 生成凭证 │ │ │ └─────────────────────────────────────────────────────────────────┘ `

3.2 流程详细说明

#### 步骤1：采购对账发现扣款项
  • 采购人员与供应商对账
  • 发现品质问题/迟延等需要扣款的情况
  • 评估扣款金额是否需要走现金流程
#### 步骤2：发起现金扣款申请
  • 采购人员进入系统，发起"现金扣款申请"
  • 填写信息：
  • 供应商（必填）
  • 扣款方向（供应商→我方 / 我方→供应商）
  • 扣款金额（必填）
  • 扣款原因（必填）
  • 关联单据（必填，原应付发票）
  • 上传扣款协议（如有）
  • 提交审核
#### 步骤3：财务审核
  • 财务人员收到审核通知
  • 审核内容：
  • 扣款事由是否合理
  • 金额是否正确
  • 附件是否齐全
  • 通过/驳回
#### 步骤4：生成现金扣款单
  • 审核通过后，系统自动生成"现金扣款单"
  • 分配唯一单据编号（格式：CASH-YYYYMMDD-XXX）
  • 状态更新为"待确认"
#### 步骤5：现金收/付款确认
  • 财务人员执行收/付款操作
  • 选择支付方式：现金/银行转账
  • 确认收/付款信息
  • 提交确认
#### 步骤6：往来核销
  • 系统自动进行往来核销
  • 更新供应商应付/应收余额
  • 生成核销记录
#### 步骤7：生成凭证
  • 根据凭证模板生成凭证分录
  • 传递到总账凭证引擎
  • 完成整个流程

十、非功能性需求

10.1 性能要求

| 指标 | 要求 | |------|------| | 单据查询响应时间 | ≤ 500ms | | 单据提交响应时间 | ≤ 1s | | 批量导出 | 支持导出1000条记录 |

10.2 安全要求

  • 所有操作记录操作日志
  • 敏感操作（如删除）需二次确认
  • 审核记录永久保存，不可删除

10.3 各国差异适配

| 国家 | 特殊要求 | 实现方式 | |------|----------|----------| | 越南 | 必须通过现金处理 | 必填扣款协议附件 | | 泰国 | 需双方书面确认 | 支持附件上传 | | 菲律宾 | 无特殊要求 | 正常流程 | | 沙特 | VAT合规 | 与VAT模块联动 | ---
