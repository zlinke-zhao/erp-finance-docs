1. 模块定位与业务目标

1.1 一句话定义

实收收款模块是ERP系统中"资金流入"的核心枢纽，负责从银行资金到账开始，经过收款登记、应收核销、凭证生成，最终写入总账的全流程管理。

1.2 在整体流程中的位置

`` 销售订单 → 送货单 → 应收对账 → 税务开票 → 发票录入 ─┐ │ 采购申请 → 采购订单 → 收货验收 → 应付对账 ──────────┤ ▼ ┌──────────────┐ │ 🔄 实收收款 │ ◄── 本模块 │ (本规格书) │ └──────┬───────┘ │ ▼ ┌──────────────┐ │ 🧾 总账凭证 │ ← P0待建 │ │ │ 借:银行存款 │ │ 贷:应收账款 │ └──────────────┘ `

1.3 核心业务目标

| 目标 | 指标 | 当前痛点 | |:---|:---|:---| | 收款可追溯 | 每笔收款100%关联到具体应收单据 | 手工Excel对账，容易漏记 | | 核销准确率 | 自动核销匹配率≥90%，人工确认率≤10% | 全靠人眼逐笔核对 | | 账龄准确性 | 应收余额与实际回款实时同步 | 月末才调一次，平时不准 | | 预收透明度 | 预收余额实时可见、自动抵扣 | 预收款记在脑子里或散落各处 | | 凭证自动化 | 收款审核通过后自动生成凭证 | 手工做分录，容易出错 | | 到账时效 | 从银行到账到系统登记≤4小时 | 发现晚、登记不及时 |

1.4 用户角色

| 角色 | 权限范围 | 典型操作 | |:---|:---|:---| | 财务会计 | 创建/编辑/删除收款单 | 录入收款、核销操作、查看报表 | | 财务主管 | 审核所有收款单、审批异常 | 审核、红冲、权限审批 | | 出纳 | 仅录入和查询 | 登记到账信息（不可核销） | | 销售经理 | 只读查看自己客户的收款情况 | 回款进度、客户欠款 | | 系统管理员 | 配置科目模板、核销规则、银行账户 | 参数配置 | ---

` POST /api/v1/receipt # 创建收款单 GET /api/v1/receipt/{id} # 查询收款单详情 PUT /api/v1/receipt/{id} # 编辑收款单(仅草稿状态) DELETE /api/v1/receipt/{id} # 删除收款单(仅草稿状态) GET /api/v1/receipt?page=&size=&status= # 分页查询列表(支持筛选)

5.2 核销操作

` POST /api/v1/receipt/{id}/writeoff # 执行核销 DELETE /api/v1/receipt/writeoff/{wo_id} # 反核销(撤销) GET /api/v1/receipt/{id}/pending-ar # 获取待核销应收列表 GET /api/v1/receipt/smart-match?receipt_id={id}&amount=50000 # 智能推荐核销方案

核销请求体示例(支持一对多):

{ "writeoff_items": [ { "ar_invoice_id": 8001, "amount": 35000.00 }, { "ar_invoice_id": 8002, "amount": 15000.00 } ], "use_prepay": false, "remark": "客户一次性付清两笔订单" } `

5.3 审批流程

` POST /api/v1/receipt/{id}/submit # 提交审核 POST /api/v1/receipt/{id}/approve # 审核通过 POST /api/v1/receipt/{id}/reject # 审核驳回 POST /api/v1/receipt/{id}/cancel # 作废(红字冲销) `

5.4 预收款

` GET /api/v1/prepay/customer/{customer_id} # 查询客户预收余额 GET /api/v1/prepay/customer/{customer_id}/transactions # 预收流水明细 POST /api/v1/prepay/deduct # 手动预收抵扣操作 `

5.5 报表

` GET /api/v1/report/collection-summary?period=month&date=2026-04 GET /api/v1/report/ar-aging?as_of_date=2026-04-14 GET /api/v1/report/collection-by-customer?from=2026-01-01&to=2026-04-14 GET /api/v1/report/collection-daily?month=2026-04 `` ---

| 集成方向 | 对接模块 | 集成内容 | 方式 | |:---|:---|:---|:---| | 上游 | 销售管理(SO) | 销售订单发货后自动产生应收单 | 事件驱动/API回调 | | 上游 | 发票管理 | 发票开具后更新应收金额 | API调用 | | 下游 | 总账引擎(GL) | 收款审核通过后自动生成凭证 | 同步调用/消息队列 | | 并行 | 银行管理(BA) | 银行账户字典、银企接口 | 共用数据/接口 | | 并行 | 客户管理(CM) | 客户信用额度、预收余额 | 共用数据/读写 | | 下游 | 票据管理(BM) | 承兑汇票收款走特殊流程 | 条件路由 | ---

7. 非功能性需求

| 维度 | 要求 | |:---|:---| | 性能 | 单次核销响应 < 2秒; 列表查询 < 500ms | | 并发 | 支持5个财务人员同时操作不冲突 | | 数据一致性 | 核销操作必须在事务中完成; 收款金额+核销总额+预收余额 三者必须平衡 | | 安全性 | 收款单审核后不可编辑; 反核销需要独立权限; 金额变更留痕 | | 审计追踪 | 所有核销/反核销操作记录操作人、时间、IP、前后值 | | 可用性 | 核销操作失败必须可重试; 不允许出现重复核销 | ---

8. 测试用例要点

| 用例类别 | 测试点 | 预期结果 | |:---|:---|:---| | 正常流程 | 1对1全额核销 | 应收余额清零,收款单状态=已核销 | | 正常流程 | 部分核销(收款<应收) | 应收减少,剩余未收>0 | | 边界测试 | 尾差¥0.05 | 自动归入财务费用 | | 边界测试 | 核销金额恰好等于应收 | 余额精确为0 | | 异常测试 | 重复核销同一张应收 | 系统拦截,提示"已核销" | | 异常测试 | 核销金额超过未收余额 | 校验拦截,"核销金额超限" | | 异常测试 | 已审核收款单编辑 | 权限拦截,"已审核单据不可修改" | | 预收场景 | 有预收时新建应收 | 弹窗提示"该客户有预收¥XXX,是否使用?" | | 预收场景 | 预收不足抵扣全部应收 | 部分抵扣+部分正常核销 | ---
