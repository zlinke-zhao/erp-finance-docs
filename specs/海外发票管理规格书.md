## 1. 模块概述

### 1.1 业务背景

公司业务覆盖东南亚及中东4个国家，需处理海外进项（供应商开给我们的发票）和销项（我们开给客户的发票）两类场景。海外发票与国内增值税发票存在本质差异：无统一税局API验真、各国税制独立、多币种处理、"票随货走"等特殊业务模式。

### 1.2 模块定位

本模块是 `发票查验模块规格书.md` 的**海外扩展**，不独立存在。核心原则：

```
一个发票录入入口 → invoice_region 字段分流 → 国内走API验真 / 海外走人工审核
                                    ↓
                   审核通过后 → 应收/应付挂账 + 凭证生成（完全复用）
```

### 1.3 覆盖国家

| 国家 | 代码 | 本位币 | 标准税率 | 税号名称 | 税号格式 | 验真渠道 | 特殊模式 |
|:---|:---:|:---|:---:|:---|:---|:---|:---|
| 🇻🇳 越南 | VN | VND | 10% | MST | 10位数字 | GDT电子平台(2025.6起) | 票随货走 |
| 🇹🇭 泰国 | TH | THB | 7% | VAT/TIN | 13位数字 | 税局网站(无公开API) | 票随货走 |
| 🇵🇭 菲律宾 | PH | PHP | 12% | TIN(BIR) | 9-12位数字 | 无统一API | — |
| 🇸🇦 沙特 | SA | SAR | 15% | VAT Number | 15位(3XX-XXXXXXXXX-X) | ZATCA/FATOORAH | 电子发票全覆盖 |

### 1.4 核心能力

| 能力 | 描述 | 优先级 |
|------|------|--------|
| 🌏 多国发票录入 | 按4国合规格式录入，自动匹配发票模板 | P1 |
| 👁️ 人工审核流程 | 附件上传、票面信息交叉比对、审核记录 | P1 |
| 💱 多币种支持 | 外币金额 + 汇率 + 本位币折算 | P1 |
| 📦 票随货走跟踪 | 越南/泰国场景：发票与物流绑定、中途遗失补票 | P1 |
| 📋 销售开票 | 按客户所在国合规格式生成销售发票 | P1 |
| 🖼️ 影像存档 | 扫描件/原件照片上传、OCR辅助录入 | P1 |
| 🔌 验真API预留 | 沙特ZATCA/越南GDT未来对接接口预留 | P2 |
| 📊 跨境发票台账 | 按国家/币种/供应商维度的台账与统计 | P2 |

## 2. 与国内发票模块的关系

### 2.1 共享部分（完全复用）

| 共享内容 | 说明 |
|:---|:---|
| `invoice_verification_log` 主表 | 通过 `invoice_region` 字段区分 |
| 查验台账查询API | 按region过滤即可 |
| 异常池表 `invoice_exception_pool` | 海外发票异常同样入池 |
| 审核通过后的挂账逻辑 | 应收/应付模块无需感知发票来源 |
| 凭证模板 | 总账凭证引擎通过event_key区分 |

### 2.2 差异部分（本模块新增）

| 差异点 | 国内发票 | 海外发票 |
|:---|:---|:---|
| **验真方式** | 腾讯云API自动 | 人工审核 + 附件比对 + 未来API预留 |
| **发票编号** | 税局统一编码 | 企业自编号 |
| **发票类型** | 12种票种代码 | 各国自定义类型 |
| **币种** | CNY | VND/THB/PHP/SAR + 汇率 |
| **票面信息** | API返回结构化数据 | 手工录入 + OCR辅助 |
| **附件** | 可选 | **必填**（扫描件/照片） |
| **票随货走** | 无 | 越南/泰国需要跟踪 |
| **销售开票** | 走金税接口 | 按国别模板生成 |

## 3. 四国发票制度详细规范

### 3.1 🇻🇳 越南 (VN)

#### 发票体系

| 维度 | 规范 |
|:---|:---|
| **发票名称** | Hóa đơn giá trị gia tăng (VAT Invoice) |
| **标准税率** | 10%（2025年7月起可能调整，需关注政策） |
| **税号名称** | MST (Mã Số Thuế) |
| **税号格式** | 10位数字，如 `0101234567` |
| **发票编号** | 电子发票：表单编号(1位) + 参考代码(6位) + 序列号(7位) = 14位 |
| **开票门槛** | 年营业额 ≥ 10亿VND 必须注册VAT |
| **管控方式** | 2025年6月起全面电子发票，通过GDT(General Department of Taxation)平台 |
| **票随货走** | ✅ 传统模式仍存在，电子化后逐步减少 |
| **验真渠道** | GDT电子平台（2025.6起可在线验证） |

#### 越南发票必须包含的信息

| 字段 | 必填 | 说明 |
|:---|:---:|:---|
| 发票编号 | ✅ | 14位电子发票编号 |
| 开票日期 | ✅ | DD/MM/YYYY 格式 |
| 卖方名称 + MST | ✅ | 卖方公司全称及10位税号 |
| 买方名称 + MST | ✅ | 买方公司全称及税号（B2B必填） |
| 商品/服务明细 | ✅ | 名称、数量、单价、金额 |
| 不含税金额 | ✅ | 小计（不含VAT） |
| VAT金额 | ✅ | 10% |
| 价税合计 | ✅ | 总金额（VND） |
| 发票类型代码 | ✅ | 1-9对应不同类型（1=标准VAT发票） |

#### 特殊场景：票随货走（越南）

```
┌─────────────────────────────────────────────────────────┐
│  越南"票随货走"业务流程                                    │
│                                                         │
│  供应商 ──发货──→ 物流 ──送货──→ 我方仓库                  │
│     │            │              │                        │
│     │         携带发票原件       │                        │
│     │         （纸质/电子打印）   │                        │
│     │                         │                        │
│     ▼                         ▼                        │
│  开票信息      物流签收确认     收货+收票                 │
│  同步传输      （附带发票）     交叉核对                  │
│                                │                        │
│                                ▼                        │
│                          票货一致？                       │
│                         /        \                      │
│                       ✅是        ❌否                    │
│                       /            \                     │
│                   录入挂账      票据异常                   │
│                                /     \                  │
│                           票丢失    金额不符             │
│                           补票流程  退回供应商            │
└─────────────────────────────────────────────────────────┘
```

**补票流程**：
1. 在ERP中标记原票"遗失"状态
2. 通知供应商重新开票
3. 收到补票后，补票编号与原单据关联
4. 补票审核通过后替换原票挂账

### 3.2 🇹🇭 泰国 (TH)

#### 发票体系

| 维度 | 规范 |
|:---|:---|
| **发票名称** | ใบกำกับภาษี (Tax Invoice / Receipt) |
| **标准税率** | 7% |
| **税号名称** | VAT Registration Number / TIN |
| **税号格式** | 13位数字，如 `0105556000123` |
| **发票编号** | 企业自编号，连续序列 |
| **开票门槛** | 年营业额 ≥ 1,800,000 THB 必须注册VAT |
| **管控方式** | 税局印发发票簿 / 电子发票并存 |
| **票随货走** | ✅ 常见模式，特别是中小企业 |
| **验真渠道** | 泰国税务局RD网站可查，但**无公开API** |
| **小企业特殊** | 年营业额 < 1,800,000 THB 可开Receipt（收据），不收VAT |

#### 泰国发票必须包含的信息

| 字段 | 必填 | 说明 |
|:---|:---:|:---|
| "ใบกำกับภาษี"字样 | ✅ | 发票标题必须包含此泰文字样 |
| 卖方名称 + 地址 + 税号 | ✅ | 13位VAT号 |
| 买方名称 + 地址 + 税号 | ✅ | B2B必填 |
| 发票编号 + 日期 | ✅ | 连续编号 |
| 商品/服务明细 | ✅ | 描述、数量、单价 |
| 不含税金额 | ✅ | 小计 |
| VAT金额 | ✅ | 7% |
| 价税合计 | ✅ | THB |
| 卖方签名/印章 | ✅ | 授权人签名 |

#### 泰国特殊：Receipt vs Tax Invoice

```
┌─────────────────────────────────────────────┐
│  供应商分类决策                                │
│                                              │
│  供应商是否注册VAT？                           │
│  ├── 是 → 开 Tax Invoice（含7%VAT）           │
│  │     → 可抵扣进项税                         │
│  │                                           │
│  └── 否 → 开 Receipt（收据）                  │
│        → 不含VAT                             │
│        → 不能抵扣进项税                       │
│        → 全额计入成本                         │
└─────────────────────────────────────────────┘
```

### 3.3 🇵🇭 菲律宾 (PH)

#### 发票体系

| 维度 | 规范 |
|:---|:---|

| **验真渠道** | **无统一API**，纯人工核验 |

| 买方信息 | ✅ | 名称、地址、TIN |
| 商品/服务明细 | ✅ | 描述、数量、单价 |

| 免税标注 | 条件 | 免税交易须标注"EXEMPT" |
| 授权签名 | ✅ | |

#### 菲律宾违规处罚（合规提醒）

### 3.4 🇸🇦 沙特阿拉伯 (SA)

#### 发票体系

| 维度 | 规范 |
|:---|:---|
| **发票名称** | Tax Invoice (فاتورة ضريبية) |
| **标准税率** | 15% |
| **税号名称** | VAT Registration Number |
| **税号格式** | 15位：`3XX-XXXXXXXXX-X`（前缀3XX + 9位识别码 + 1位校验码） |
| **发票编号** | 企业自编号（须符合FATOORAH标准） |
| **开票门槛** | 年营业额 ≥ 375,000 SAR 必须注册VAT |
| **管控方式** | **2025年起FATOORAH电子发票全覆盖**（ZATCA） |
| **票随货走** | ❌ 无此概念 |
| **验真渠道** | ZATCA平台可验，**有验证接口可研究对接** |
| **发票类型** | 简化发票(Simplified) / 标准发票(Standard) / 信用卡发票 |

#### 沙特电子发票（FATOORAH）体系

沙特是全球电子发票推进最快的国家之一，分两阶段：

| 阶段 | 时间 | 要求 |
|:---|:---|:---|
| **第一阶段** | 2021-2023 | 生成电子发票（UBL/XML格式），生成UUID |
| **第二阶段** | 2023-2025 | 集成ZATCA，发票实时上报税局，含QR Code |
| **全覆盖** | 2025年起 | 所有VAT纳税人必须使用电子发票 |

#### 沙特发票必须包含的信息

| 字段 | 必填 | 说明 |
|:---|:---:|:---|
| VAT发票字样 | ✅ | "فاتورة ضريبية" (Tax Invoice) |
| 卖方VAT号 | ✅ | 15位 |
| 买方VAT号 | ✅ | B2B必填 |
| 发票UUID | ✅ | FATOORAH生成的唯一标识 |
| 开票日期时间 | ✅ | 含时间戳 |
| 商品/服务明细 | ✅ | 含VAT商品编码 |
| 不含税金额 | ✅ | |
| VAT金额 | ✅ | 15% |
| 价税合计 | ✅ | SAR |
| QR Code | ✅ | 第二阶段要求，含TLV格式加密数据 |
| 卖方地址 | ✅ | |

#### 沙特特殊：简化发票 vs 标准发票

```
┌──────────────────────────────────────────────┐
│  发票类型决策                                  │
│                                               │
│  买方是否为VAT注册企业？                        │
│  ├── 是 → 标准发票 (Standard Tax Invoice)      │
│  │     → 包含双方完整VAT号                     │
│  │     → XML格式上报ZATCA                      │
│  │     → 买方可用于进项抵扣                    │
│  │                                            │
│  └── 否 → 简化发票 (Simplified Tax Invoice)    │
│        → 仅含卖方VAT号                        │
│        → 含QR Code                            │
│        → 终端消费者场景                        │
└──────────────────────────────────────────────┘
```

## 4. 数据模型设计

### 4.1 主表扩展：invoice_verification_log 增加字段

在现有国内发票查验表基础上增加海外字段：

```sql
-- ============================================================
-- 海外发票扩展字段（在 invoice_verification_log 上 ALTER）
-- ============================================================

-- 地区与类型
ALTER TABLE invoice_verification_log
  ADD COLUMN invoice_region        VARCHAR(5)    DEFAULT 'CN'
    COMMENT '发票地区: CN国内/VN越南/TH泰国/PH菲律宾/SA沙特/OTHER';

ALTER TABLE invoice_verification_log
  ADD COLUMN overseas_inv_type     VARCHAR(50)   DEFAULT NULL
    COMMENT '海外发票类型: VN=VAT Invoice/TH=Tax Invoice/PH=Sales Invoice/SA=Standard or Simplified';

-- 多币种
ALTER TABLE invoice_verification_log
  ADD COLUMN currency              VARCHAR(5)    DEFAULT 'CNY'
    COMMENT '币种: CNY/VND/THB/PHP/SAR/USD等';

ALTER TABLE invoice_verification_log
  ADD COLUMN exchange_rate         DECIMAL(14,8) DEFAULT NULL
    COMMENT '汇率（1外币=?CNY），取自汇率表';

ALTER TABLE invoice_verification_log
  ADD COLUMN exchange_rate_date    DATE          DEFAULT NULL
    COMMENT '汇率日期';

ALTER TABLE invoice_verification_log
  ADD COLUMN local_amount          DECIMAL(14,2) DEFAULT NULL
    COMMENT '本位币(CNY)金额 = amount_with_tax × exchange_rate';

ALTER TABLE invoice_verification_log
```

## 9. 风险控制

### 9.1 审核风控规则

| 规则 | 触发条件 | 处理方式 |
|:---|:---|:---|
| **税号格式校验** | 税号不符合该国格式 | 前端拦截，不允许提交 |
| **金额合理性** | 单张发票折算CNY > 100万 | 自动升级审批（需财务主管） |
| **同一供应商频率** | 同一供应商一周内提交 > 10张 | 标记为高风险，需主管复核 |
| **汇率偏离预警** | 手动修改汇率偏离银行中间价 > 3% | 系统警告，需备注原因 |
| **附件缺失** | 提交时未上传附件 | 前端拦截，附件为必填 |
| **重复发票检查** | 同一海外发票编号已存在 | 前端警告，允许关联但不允许重复挂账 |

### 9.2 各国合规风险提示

| 国家 | 主要风险 | 防控措施 |
|:---|:---|:---|
| 🇻🇳 越南 | 2025年电子发票新政，格式不合规可被处罚 | 跟踪GDT最新模板格式，定期更新invoice_template |
| 🇹🇭 泰国 | 小供应商开收据(Receipt)不能抵扣VAT | 系统区分Tax Invoice vs Receipt，非VAT发票全额计入成本 |
| 🇵🇭 菲律宾 | 未开票/不完整发票处罚严厉（含监禁风险） | 系统强制校验BIR要求字段，不完整不允许通过 |
| 🇸🇦 沙特 | FATOORAH电子发票要求严格，QR Code必须合规 | 沙特销项发票须符合ZATCA XML Schema，QR Code自动生成 |
